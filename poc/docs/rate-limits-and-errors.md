# Rate Limits & Error Handling

Observed during Phase 1 PoC (July 2026).

## TronGrid rate limits (documented tiers)

| Tier | Approximate limit | Notes |
|------|-------------------|-------|
| No API key (public) | ~3 req/s per IP | Aggressive throttling; not suitable for production |
| Free API key | ~15 req/s | Sufficient for single-wallet monitoring |
| Paid tiers | Higher | For multi-wallet / high-frequency use |

Source: [TronGrid documentation](https://developers.tron.network/docs/trongrid) — verify current numbers in your dashboard.

## Observed during burst test (Nile, no API key)

25 rapid sequential requests to `/wallet/getnowblock` and TRC20 endpoints: **all returned HTTP 200** with no throttling. Nile testnet is more permissive than mainnet.

**Recommendation:** always configure `TRONGRID_API_KEY` before mainnet or production-like load testing.

## Invalid input error shape

Request:
```
GET /v1/accounts/NOT_A_VALID_ADDRESS/transactions/trc20?limit=1
```

Response (HTTP 400):
```json
{
  "success": false,
  "error": "A valid account address is required.",
  "statusCode": 400
}
```

## Rate limit error shape (typical)

When throttled, TronGrid commonly returns **HTTP 429** with a JSON or plain-text body. Re-run `npm run test:rate-limits` with an API key on mainnet if you need a captured 429 sample for your environment.

Application handling (Phase 3):

- **429 / 503:** exponential backoff, retry
- **400:** log and skip (bad config)
- **Network errors:** retry with jitter, do not advance watermark

## Reconciliation interval justification

| Assumption | Value |
|------------|-------|
| Poll interval | 5 minutes (300s) |
| API calls per poll | ~1–2 |
| Daily calls | ~288–576 |
| Free tier at 15 req/s | Theoretical max ~1.3M req/day |

**Conclusion:** 5-minute reconciliation uses **< 0.05%** of daily capacity for a single wallet — conservative and safe.

Confirmation updater (every 12s) uses **local DB + one `getnowblock` call per tick** (~7,200/day) — still well within limits.
